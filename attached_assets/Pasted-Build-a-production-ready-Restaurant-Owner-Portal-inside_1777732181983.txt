Build a production-ready Restaurant Owner Portal inside our existing app for a restaurant-specific student meal package platform.

## Context

Our app allows students to buy meal packages tied to a specific restaurant and meal slot, such as:

- Restaurant A — 10 Lunch Plan
- Restaurant B — 20 Dinner Plan
- Restaurant C — 30 Full-Day Plan

Students cannot use one restaurant’s package at another restaurant. Each package belongs to one restaurant and one meal slot/type. Restaurants need their own secure dashboard to view package sales, active subscribers, upcoming meals, cancellations, and settlements.

The restaurant portal must be secure, scalable, and ready for thousands of users and multiple restaurants.

---

# 1. Goal

Create a secure, role-based Restaurant Owner Dashboard with a separate login gateway for restaurant partners.

After login, a restaurant owner should only see data belonging to their assigned restaurant or restaurants.

The portal should allow restaurant partners to:

- View active meal packages
- View subscribers per package
- View upcoming meal counts
- View locked meals for preparation
- View cancellations and no-shows
- View settlement reports
- Export reports
- Receive scheduled reports by email
- Manage limited restaurant/package information if permitted by admin

---

# 2. User roles

Implement role-based access control.

Required roles:

txt student restaurant_owner restaurant_staff admin super_admin 

## Role permissions

### restaurant_owner

Can view and manage data for assigned restaurant(s).

Can access:

txt Dashboard Packages Subscribers Upcoming Meals Cancellations Settlements Reports Restaurant Profile 

Can perform actions:

txt View package metrics View upcoming/locked meals Download reports Update limited restaurant profile fields Mark meal preparation statuses if enabled 

Cannot access:

txt Other restaurants’ data Student private payment details Admin-only financial internals Platform-wide analytics Internal commission configuration unless explicitly allowed 

### restaurant_staff

Limited access.

Can access:

txt Upcoming Meals Preparation List Order Status Updates 

Cannot access:

txt Settlements Revenue reports Subscriber personal data beyond required delivery/prep info Package financials 

### admin / super_admin

Can access everything.

Admin can assign restaurant owners/staff to restaurants.

# 3. Authentication requirements

Create a separate restaurant login gateway.

Route example:

txt /restaurant/login /restaurant/dashboard 

or mobile route:

txt RestaurantLoginScreen RestaurantDashboardScreen 

Login options:

txt Email + password Phone OTP Magic link optional 

Use existing auth system if available. If existing app uses Supabase/Firebase/custom auth, integrate with that system instead of creating duplicate auth.

On login:

1. Authenticate user.
2. Check user role.
3. Check restaurant assignment.
4. If user is not restaurant_owner, restaurant_staff, admin, or super_admin, block access.
5. Redirect unauthorized users to a clear error screen.

Unauthorized message:

txt You do not have access to the restaurant portal. Please contact support if this is a mistake.

# 4. Data access security

This is critical.

Every query must be scoped by restaurant access.

A restaurant owner must only fetch data for restaurant IDs assigned to them.

Create or use a mapping table:

txt restaurant_users 

Suggested fields:

txt id user_id restaurant_id role: owner / staff / manager status: active / invited / suspended created_at updated_at 

Backend must enforce:

txt current_user can only read restaurant data where restaurant_users.user_id = current_user.id and restaurant_users.status = 'active' 

Do not rely only on frontend filtering. Enforce access control at backend/database/API level.

If using Supabase, add RLS policies for all restaurant portal tables/views.

If using custom API, validate restaurant access in middleware before every request.

# 15. Temporary closure / holiday mode

Restaurant owners must be able to mark unavailable dates.

Feature:

txt Holiday Mode / Temporary Closure 

Fields:

txt Start date End date Meal slots affected: lunch / dinner / both Reason 

When restaurant marks closure:

1. New scheduling should be blocked for those dates.
2. Existing scheduled meals should be flagged.
3. Admin should be notified.
4. Affected students should receive notification.
5. Credits should be restored or meals rescheduled based on policy.

Do not silently cancel meals. That is how support tickets spawn like mosquitoes.

# 19. Row-level security / access policy

txt Restaurant owner can select rows only where restaurant_id is assigned to auth.uid() 

Example logic:

# 26. Critical business logic

## Meal count logic

For tomorrow’s meals:

txt expected_meals = active scheduled meals - free cancellations before cutoff 

After cutoff:

txt locked_meals = scheduled meals not cancelled before cutoff 

## Payout logic

For each delivered meal:

txt restaurant_payout = meal_value - platform_commission 

For late cancellation:

txt late_fee = meal_value * late_cancellation_percentage restaurant_share = late_fee * restaurant_late_fee_share platform_share = late_fee - restaurant_share 

For restaurant cancellation:

txt restaurant_payout = 0 user meal restored possible penalty later 

For no-show:

txt restaurant_payout eligible full meal deducted

# 29. Acceptance criteria

This feature is complete only when:

txt Restaurant owners can log in through a separate gateway. Only approved restaurant users can access the portal. Every restaurant user only sees assigned restaurant data. Dashboard metrics load correctly. Packages, subscribers, upcoming meals, cancellations, and settlements are visible. Date and meal-slot filters work. CSV export works and respects access control. Email report settings work or are safely stubbed. All status-changing actions create audit logs. Unauthorized access attempts are blocked. UI handles loading, empty, and error states. Performance is acceptable with large datasets. 

---

# 30. Important implementation note

Do not build this as a frontend-only dashboard.

The restaurant portal must be backed by secure APIs or database policies.

All financial calculations, settlement calculations, cancellation deductions, and access permissions must be calculated/enforced on the backend.

Frontend should display trusted backend results, not calculate business-critical money logic itself.

Build this like production infrastructure, not a pretty admin panel.
